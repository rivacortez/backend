## Ventana de fechas y moneda
- Query opcional **`?from=ISO&to=ISO`** (ISO 8601 con offset). Si faltan, la ventana es **hoy** en la zona del tenant: `from` = medianoche local (**America/Lima**, UTC-5 sin DST), `to` = `now`. Validación `from <= to` → si no, **400**.
- Lógica pura y testeable en `src/reports/report-window.util.ts` (`resolveWindow`, `startOfLimaDay`, `limaDayKey`, `lastNLimaDays`); cubierta por `report-window.util.spec.ts`.
- **Toda la moneda** se devuelve como **string** `.toFixed(2)` (PEN). Cálculos con `Prisma.Decimal` (Prisma 6).
- **Base de todos los agregados:** ventas **emitidas** (`Sale.status='issued'`) con `issuedAt ∈ [from, to]`. Las anuladas (`void`) no suman (sólo se cuentan donde aplica). Las unidades/revenue por plato provienen de los `order_items` vivos (`deletedAt=null`) de las órdenes de esas ventas.

## HU-07-03 · Dashboard de cajero — `GET /api/reports/dashboard/cashier`
```gherkin
GIVEN cajero con turno abierto
WHEN abre su dashboard
THEN ve: total cobrado hoy, desglose por metodo, tickets pendientes, anulaciones
AND el detalle se actualiza tras cada cobro
```
- `read Sale` (staff incluido). Agrega la caja del día (o de la ventana):
```jsonc
{
  "date",                 // ISO del inicio de la ventana
  "salesCount",           // nº de tickets emitidos
  "voidCount",            // nº de tickets anulados en la ventana
  "totalCollected",       // Σ total de los emitidos (string)
  "byMethod": { "cash", "card", "yape", "plin" },  // Σ payment.amount por método (strings)
  "avgTicket"             // totalCollected / salesCount (0 si no hay ventas)
}
```
> "Tickets pendientes" del Gherkin se interpreta como las **anulaciones** (`voidCount`) y el detalle por método; el flujo de cobro de E04 cierra la orden al pagar (no quedan tickets "a medias"). `byMethod` siempre trae las 4 claves.

## HU-07-01 · Dashboard de admin (ejecutivo) — `GET /api/reports/dashboard/admin`
```gherkin
GIVEN admin autenticado
WHEN abre su dashboard
THEN ve widgets: ventas hoy/semana/mes, food cost %, margen, top 5 platos, alertas activas
```
- `read Report`. KPIs financieros + operativos:
```jsonc
{
  "date",
  "revenueToday",
  "revenue7d",         // Σ total emitidos en los últimos 7 días locales (Lima)
  "ordersToday",
  "avgTicket",         // revenueToday / ordersToday
  "grossMarginPct",    // (Σ subtotal − COGS) / Σ subtotal · 100 (período = hoy)
  "topDishes": [ { "name", "qty", "revenue", "contribution" } ],  // top 5
  "lowStockCount",
  "salesByDay7d": [ { "day": "YYYY-MM-DD", "revenue" } ]   // 7 días asc; días sin venta = 0
}
```
- **`grossMarginPct`** = margen bruto sobre el subtotal (sin IGV) de las ventas del período: `(Σ subtotal − COGS) / Σ subtotal · 100`, con **COGS** = `Σ por plato (costoIngredientes/unidad · unidades vendidas)`. El costo de ingredientes por unidad se obtiene del **BOM recursivo** (`RecipesService.costPerYieldTx`, reutilizado de E06).
- **`contribution`** por plato del top = `revenue − (costoIngredientes/unidad · qty)`.
> "Ventas semana/mes" se materializa con `revenue7d` + `salesByDay7d` (la serie diaria habilita el gráfico de los 7 días); el mes completo y "food cost %" agregado se cubren en E06 (`/api/costing/*`) y se ampliarán en Inc 2 (HU-07-06). "Personalizar layout (drag-and-drop)" = frontend.

## HU-07-04 · Reporte de ventas — `GET /api/reports/sales?from=&to=&groupBy=day|method|docType`
```gherkin
GIVEN gerente en modulo de reportes
WHEN selecciona "Ventas" con rango de fechas, mesero, mesa, tipo de doc
THEN se genera reporte con: tickets, items vendidos, totales, comparativo vs periodo anterior
AND se exporta a PDF/Excel/CSV
```
- `read Report`. `groupBy` ∈ {`day` (default), `method`, `docType`}:
```jsonc
{
  "from", "to",
  "totalRevenue", "salesCount", "avgTicket",
  "byMethod":  { "cash", "card", "yape", "plin" },
  "byDocType": { "boleta", "factura" },
  "series": [ { "key", "revenue", "count" } ]   // según groupBy
}
```
- `series`:
  - `groupBy=day` → una entrada por **día local (Lima)** con venta, ascendente; `count` = nº de tickets de ese día.
  - `groupBy=method` → 4 entradas (cash/card/yape/plin); `revenue` = `byMethod[key]`; `count` = nº de pagos de ese método (un ticket mixto cuenta en cada método que usó).
  - `groupBy=docType` → boleta/factura; `count` = nº de tickets de ese tipo.
> **Filtros mesero/mesa** y **comparativo vs período anterior** se difieren (refinamiento de Inc 2); el rango de fechas + agrupaciones cubren el núcleo. La **exportación** (HU-07-10) es Inc 2 (R2).

## HU-07-08 · Análisis Pareto de platos — `GET /api/reports/pareto-dishes?from=&to=`
```gherkin
GIVEN ventas del periodo
WHEN abro analisis Pareto
THEN veo grafico con platos ordenados de mayor a menor revenue
AND linea acumulada del 80%
AND identifico Stars, Plowhorses, Puzzles, Dogs (Menu Engineering)
```
- `read Report`. Clasificación **ABC** por revenue acumulado (80/20):
```jsonc
{
  "items": [
    { "name", "qty", "revenue", "revenuePct", "cumulativePct", "abcClass": "A|B|C" }
  ],
  "totalRevenue"
}
```
- Platos ordenados **desc por revenue** (desempate por nombre para estabilidad). `revenuePct = revenue/totalRevenue·100`; `cumulativePct` = suma acumulada **ascendente** que cierra en ~100.
- **ABC:** `A` mientras `cumulativePct ≤ 80`; `B` mientras `≤ 95`; `C` el resto. (Equivalente operativo del 80/20 / Menu Engineering del Gherkin; la matriz Stars/Plowhorses/Puzzles/Dogs completa, que cruza popularidad × margen, se difiere.)
