## Modelo de datos (RLS FORCE en ambas tablas — riesgo R4)

- **`sales`** (`Sale`): `orderId` FK→`orders` **único** (un ticket por orden), `serie`, `number` (correlativo), `docType` (`boleta|factura`), `customer?`, `customerDoc?`, `subtotal`/`igv`/`total` `Decimal(12,2)`, `status` (`issued|void`, default `issued`), `voidReason?`, `issuedAt`. `@@unique([tenantId, serie, number])` (correlativo por tenant+serie) y `@@unique([orderId])`.
- **`payments`** (`Payment`): `saleId` FK→`sales` `ON DELETE CASCADE`, `method` (`cash|card|yape|plin`), `amount` `Decimal(12,2)`. **Varios pagos por venta** → pago mixto (HU-04-06).

Migración `sales_payments`; bloque RLS FORCE (`ENABLE`+`FORCE ROW LEVEL SECURITY` + policy `tenant_isolation`) **apendizado** para ambas tablas. `tenant_id` siempre desde el JWT; todo el acceso vía `runInTenant`.

## Cálculo de impuestos (precios INCLUYEN IGV)

Los precios de la carta ya incluyen IGV (mercado peruano). Para una orden:

- `total = Σ (unitPrice · qty)` de sus ítems vivos.
- `subtotal = total / (1 + igvRate)` (del `tenant.igvRate`, default `0.18`).
- `igv = total − subtotal`.

Toda la moneda se devuelve como **string** `.toFixed(2)` (PEN). Ej.: ítem 118.00 → subtotal 100.00, IGV 18.00, total 118.00.

## HU-04-02 · Generar cuenta final (ticket) + HU-04-04/05/06 · Pagos

```gherkin
GIVEN pre-cuenta lista
WHEN cajero selecciona tipo (BOLETA/FACTURA) y datos del cliente
THEN se asigna serie y correlativo segun tipo AND el ticket queda listo
GIVEN ticket pendiente WHEN cajero ingresa pago(s) (efectivo / Yape / Plin / tarjeta)
THEN se valida que cubran el total AND el ticket pasa a PAID
```

**`POST /api/orders/:id/pay`** · `create Sale` · `@Audited('sale.pay')`. Body:

```jsonc
{ "payments": [{ "method": "cash|card|yape|plin", "amount": number>0 }],  // min 1
  "docType": "boleta|factura",  // default boleta
  "customer"?: string, "customerDoc"?: string }
```

Reglas (una sola transacción `runInTenant`):

1. La orden debe estar `open`/`served` y **no** pagada (si ya pagó/anuló → **409**).
2. `Σ amount ≥ total` (si no → **400**).
3. `serie = docType==='factura' ? 'F001' : 'B001'`; `number = (max number para ese tenant+serie) + 1` (correlativo, arranca en 1).
4. Crea `Sale` + N `Payment`; `order.status='paid'`; libera la mesa (`table.status='free'`).

Devuelve `{ order: OrderView, sale: SaleView }` (mismo contrato del mock BFF `orders/[id]/pay.post.ts` → adaptador trivial).

> Pago electrónico (HU-04-05): se **registra** el método; **no hay pasarela** (Yape/Plin/tarjeta sin gateway). La "referencia/nº de operación" del Gherkin no se modela en Inc 1.
> Efectivo (HU-04-04): el **vuelto** lo calcula el frontend (recibido − total); no se persiste.
> Mixto (HU-04-06): varios `Payment` cuya suma cubre el total.

## Refinamiento QA-02 (bugfix, reporte QA usuario final pre-demo) — Descuento en la cuenta

> **Estado:** 🟢 hecho. Sin nuevo número de HU (mismo criterio que el refinamiento `HU-05-consumo-en-venta`).

**Root cause:** el descuento **no existía en el backend en absoluto** — ni campo de persistencia en `Order`, ni endpoint, ni participación en el cálculo de totales de `pre-bill`/`pay`/`split`. El frontend (`DiscountSheet.vue`) calculaba el descuento SOLO como preview local (`total − discount`) y su BFF (`server/utils/pos-adapter.ts`) **descartaba el campo `discount` a propósito** (comentario explícito: _"descuento (discount) aún no existe en el backend → se ignora"_). El toast "Descuento aplicado" era optimismo de UI sin persistencia — el cobro salía siempre por el 100% del bruto.

**Modelo de datos (`orders`/`sales`, migración `discount_fields`, sin tablas nuevas):**

- `orders` gana `discount_type` (`pct|amount`, nullable), `discount_value` `Decimal(12,2)` nullable, `discount_reason` nullable. Los 3 campos viven/mueren juntos (únicos escritores: `applyDiscount`/`removeDiscount`).
- `sales` gana los mismos 3 campos + `discount_amount Decimal(12,2) default 0` — **snapshot inmutable** del descuento AL MOMENTO DEL COBRO (el ticket ya emitido no debe mutar si la orden cambiara después).

**Endpoints nuevos (`pos` — `OrdersController`):**

- `POST /api/orders/:id/discount` · `update Sale` (**manager/owner**; staff → **403** — mismo criterio que anular ticket HU-04-07, coincide con el badge "Owner" de la UI) · `@Audited('order.apply_discount')`. Body `{ type: 'pct'|'amount', value: number≥0, reason: string (obligatorio) }`. `pct` acotado 0-100 (Zod); `amount` no puede exceder el bruto vivo de la orden (400 si excede). 409 si la orden ya está `paid`/`void`.
- `DELETE /api/orders/:id/discount` · mismo gate · `@Audited('order.remove_discount')`. Limpia los 3 campos.
- `OrderView` gana `discount: { type, value, reason } | null` (display; el cómputo monetario autoritativo vive en `OrdersService.computeDiscountAmount`, reutilizado por `billing`).

**Cálculo de totales (`billing` — `BillingService.computeTotals`, ÚNICA fuente de verdad para pre-bill/pay/split):**

```
grossTotal = Σ unitPrice·qty (bruto, antes de descuento)
discountAmount = computeDiscountAmount(order, grossTotal)   // 0 si no hay descuento
  · pct:    grossTotal · value/100 (2dp)
  · amount: min(value, grossTotal)                          // nunca deja el total negativo
total = grossTotal − discountAmount                          // esto es lo que se cobra
subtotal = total/(1+igvRate); igv = total − subtotal         // IGV sobre la base YA descontada
```

`PreBillView`/`SaleView`/`SplitView` ganan `grossTotal` + `discount: { type, value, reason, amount } | null` (aditivo, no rompe consumidores existentes). `SaleView.discount`/`grossTotal` derivan del **snapshot persistido** en `Sale` (no se recalculan desde la orden).

**División de cuenta (`split`) también respeta el descuento** (root cause incluía "dividir por persona" repartiendo el bruto):

- `mode=equal`: divide `totals.total` (ya descontado) — cambio automático al enrutar por `computeTotals`.
- `mode=items`: si NO hay descuento, cada share = su bruto exacto (0 cambio de contrato). CON descuento, reparte el descuento **proporcional al peso bruto de cada parte** (redondeo hacia abajo por parte, resto en la 1ª — mismo criterio que `splitEqual`) → `Σ shares.total === totals.total` exacto, siempre.

**Casos límite cubiertos (pedidos explícitamente por el QA):** 0% (cuenta intacta, `discountAmount="0.00"`), 100% (cuenta gratis, `total/subtotal/igv="0.00"`, se puede cobrar con un pago nominal), `amount` que excede el bruto → 400, orden sin descuento nunca aplicado → `discount: null` en todo el flujo (regresión).
